## 1. Точки исчезают при первой загрузке карты

**Причина (гипотеза).** В `TbilisiMap.tsx` начальные данные подставляются в источник `churches` двумя путями: один раз в обработчике `map.on("load")` через `filteredRef.current`, и параллельно — в `useEffect([filtered])` через `map.once("idle", apply)`. При определённой последовательности (rows приходят до `load`, эффект ставит `once("idle")` ещё до того, как сам источник создан в `load`-колбэке; либо `idle` срабатывает в неожиданном порядке) в источнике остаётся пустая `FeatureCollection`. После любого изменения фильтров эффект гарантированно перезаписывает данные — поэтому «Сбросить фильтры» оживляет карту.

**Что сделать.**
- Ввести состояние `mapReady` (выставляется в `true` внутри `map.on("load")` после `addSource/addLayer`).
- Удалить `setData` из самого `load`-колбэка.
- В `useEffect([filtered, mapReady])` делать единственное действие: если `mapReady && map.getSource("churches")` → `src.setData(...)`. Никаких `once("idle")`.
- Это убирает гонку: данные всегда применяются после готовности слоя и реактивно следуют за `filtered`.

## 2. Перепроверка координат точек

Сейчас часть точек проставлена «по центру района» (`confidence: low_district` / `low_approx`). Пройти умной проверкой по двум явно указанным и по всем низкоточным:

- **№57 «Св. Георгия (Сабуртало)»** — текущая `41.725, 44.75`. По уличной топонимике церковь Св. Георгия в Сабуртало традиционно привязывают к району ул. Пекини / Важа-Пшавела. Проверить через web-поиск (груз. ка-тексты + OSM `nominatim`) и обновить координаты с повышением `confidence` до `medium`/`high`.
- **№139 «Григория Богослова (Глдани)»** — текущая `41.79, 44.82`. Надо проверить, действительно ли это южная часть Глдани или северная (микрорайоны Гldani-Nadzaladevi vs. собственно Гldani-massive). Сверить с OSM/груз. источниками.
- Заодно прогнать остальные ~10–15 точек с `confidence in {low_district, low_approx}` через web-поиск + Nominatim: где находится подтверждение (улица/площадь), сдвинуть координату и поднять `confidence`. Где нет — оставить как есть, но добавить пометку.

Все правки координат — в `public/data/tbilisi-churches.json` через одноразовый Python-скрипт (фиксируем правки списком, чтобы было воспроизводимо).

## 3. Многоязычность полей «История» и «Примечание»

Сейчас `historicalNote` и `note` в `public/data/tbilisi-churches.json` — простые строки только на русском. Нужно:

- В `src/lib/tbilisiChurches.ts` поменять типы:
  ```ts
  historicalNote: string | { ru: string; en: string; ka: string };
  note:           string | { ru: string; en: string; ka: string };
  ```
  (string оставляем как fallback на время миграции).
- В `TbilisiMap.tsx` в карточке выбранной точки выбирать значение по текущему `lang`: `typeof x === "string" ? x : (x[lang] || x.ru)`.
- В JSON преобразовать оба поля во все 108 записей в объекты `{ru, en, ka}`. Перевод — батчем через скрипт с моделью `google/gemini-2.5-flash` (Lovable AI Gateway), сохраняя термины (имена церквей, районы) консистентно с уже существующими `name.en` / `name.ka`. Сначала генерируем перевод, затем вручную (скриптом) дописываем в JSON.

## 4. Чистка артефактов «См. № N»

В 8 записях `historicalNote` содержит только ссылки вида `См. № 65`, `См. № 132` и т.п. (id 83, 85, 125, 127, 128, 129, 131, 133). Это следы старого реестра дублирующих метрических книг (венчание/смерть отдельной книгой при той же церкви).

Подход:
- Для каждой такой записи попытаться восстановить смысл из связки «эта запись» ↔ «запись с указанным №» (как правило это та же церковь, но другая метрическая книга). Если связка очевидна — заменить на осмысленный текст («Метрическая книга бракосочетаний/смертей, относится к той же церкви, что и №65 …») во всех трёх языках.
- Если восстановить не удаётся — просто **очистить** поле (`""` во всех трёх языках), чтобы артефакт не показывался.

5. Проверка в превью: точки появляются сразу; карточки на en/ka показывают переведённые «История» и «Примечание»; артефактов «См. № …» больше нет.
