## Что меняем

Сейчас AI-геокодер умеет только одно: создать `pending`-предложение в `coord_suggestions`. Он не смотрит в `parishes.geojson`, и если найденная точка попадает рядом с уже опубликованным селом — она:
- либо проскакивает мимо и создаёт визуальный дубликат после одобрения,
- либо блокируется анти-дублем только в том случае, если рядом висит другое `pending` предложение (как с «Анагой»).

Добавим третий исход: **авто-слияние с опубликованной точкой** через `feature_overrides` (`action='edit'`, `published=true`) — ровно тот же механизм, что мы вручную применяли для Биа/Джолеви/Тетрицкаро.

## Поведение после изменений

Для каждого кандидата от AI:

1. **Поиск ближайшей опубликованной точки** в `parishes.geojson` в радиусе `mergeRadiusM` (по умолчанию **1500 м**, отдельный ползунок в UI).
2. **Проверка «это то же село»** — должны выполняться все три условия:
   - `isProbableMatch` по названию села (RU + EN, уже есть в `fuzzyMatch.ts`).
   - `normalizeAdmin` по уезду ИЛИ по региону совпадает (т.е. «Тифлисский уезд» = «Тифлисский район», «Зугдиди» = «Зугдидский уезд»). Достаточно совпадения хотя бы одного из двух полей — это покрывает кейс «вместо района указан тот же уезд».
   - `confidence` AI ≥ `minMergeConfidence` (по умолчанию **0.75**, отдельный ползунок).
3. **Если совпало** → авто-слияние:
   - Берём существующую `FeatureData` (с применёнными оверрайдами).
   - **Годы:** `startYear = min`, `endYear = max`, `yearsRaw` — объединение через запятую с дедупликацией диапазонов (та же логика, что в ручных мерджах).
   - **Церковь:** если пустая — заполняем из unlocated; если непустая и отличается — добавляем через ` / ` (на всех трёх языках).
   - **Уезд/регион:** оставляем как есть в опубликованной точке (она канон), но если у каноники поле пустое, а у unlocated заполнено — подставляем.
   - **Координаты:** не трогаем (опубликованная точка — канон).
   - Создаём/обновляем запись в `feature_overrides` (`action='edit'`, `published=true`, `notes='AI auto-merge: <причина>, OSM <display_name>'`).
   - В UI лог: новый статус `merged` (зелёная иконка слияния) + ссылка на feature_id.
4. **Если рядом точка есть, но проверка «то же село» провалилась** → текущее поведение `skipped` с пометкой «конфликт с опубликованной точкой «X» (Y м, причина: имя/уезд расходятся)» — чтобы админ глянул вручную.
5. **Если рядом ничего нет** → текущее поведение: `inserted` в очередь `coord_suggestions` как `pending`.

Проверка против других `pending` в `coord_suggestions` остаётся (это то, что сейчас зовётся `conflictRadiusM`).

## Изменения в коде

- **`src/lib/aiGeocoder.functions.ts`**
  - Подгрузить `parishes.geojson` (через `import` как `unlocated.json`) **и** список оверрайдов через `supabaseAdmin.from('feature_overrides').select(...)`. Слепить применённую версию каждого feature в памяти один раз на запуск handler-а.
  - Построить spatial-индекс по сетке (bucket 0.05°) для быстрого поиска кандидатов в радиусе — иначе для 1355 точек O(N) на каждое предложение.
  - Новая функция `findMergeTarget(item, lat, lon, features, opts)` — возвращает `{ featureId, reason } | null`.
  - Новая функция `buildMergeOverride(existing, item, osmName)` — собирает обновлённую `FeatureData` (годы, церковь, уезд по правилам выше).
  - В основном цикле: после `validateOsmMatch`, **перед** проверкой `coord_suggestions`, сначала пробуем `findMergeTarget`. Если есть — пишем `feature_overrides` и логируем `merged`. Если нет — текущая ветка.
  - Расширить `BatchResult.log[].status` типом `"merged"`, добавить поле `featureId?: number`.
  - В `inputValidator` добавить `mergeRadiusM` (default 1500, 0..5000) и `minMergeConfidence` (default 0.75, 0.5..1).

- **`src/components/admin/AiGeocoderPanel.tsx`**
  - Два новых ползунка: «Радиус авто-слияния» и «Мин. уверенность для слияния».
  - В сводке счётчиков — новый чип `слито: N` (зелёный, иконка `GitMerge` из lucide).
  - В строке лога — иконка `GitMerge` для статуса `merged`, текст «слито с feature #ID: <название>».
  - В шапке-инструкции — короткое описание новой логики (1-2 предложения).

- **Тип `BatchResult`** в файле `aiGeocoderPanel.tsx` — добавить `merged: number` и `"merged"` в union статусов.

## Что НЕ меняем

- Не трогаем `coord_suggestions` flow для случаев «вокруг чисто» — он работает.
- Не убираем существующий `conflictRadiusM` — это другая проверка (антидубль внутри очереди модерации).
- Не делаем авто-`delete` оверрайдов на дубли в `parishes.geojson` — этим занимается отдельный ручной процесс дедупликации.

## Дефолты (можете поправить до старта)

| Параметр | Дефолт | Что значит |
|---|---|---|
| `mergeRadiusM` | 1500 м | Радиус поиска опубликованной точки для слияния |
| `minMergeConfidence` | 0.75 | Ниже — только в `pending`, без авто-мерджа |
| Совпадение admin | уезд ИЛИ регион | Достаточно одного для подтверждения «то же село» |
| Расхождение церкви | append через ` / ` | Не теряем данные о другой церкви |
| Расхождение годов | union (min..max) | + объединённый `yearsRaw` |

Если хотите более консервативно (например, требовать совпадения **и** уезда и региона, или радиус 500 м, или порог 0.85) — скажите перед стартом, поправлю в плане.
